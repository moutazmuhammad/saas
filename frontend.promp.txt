================================================================================
                    SAAS PLATFORM — FRONTEND DESIGN PROMPT
================================================================================

PROJECT GOAL
------------
This is a multi-tenant SaaS platform built on Odoo 18. It allows businesses to
browse cloud-hosted Odoo services (ERP, CRM, Accounting, etc.), choose a plan
(free trial or paid), provision a fully isolated Odoo instance in a Docker
container, and manage it through a customer portal. The platform handles the
entire lifecycle: registration with OTP verification, service discovery, instance
provisioning, billing with proration, plan upgrades/downgrades, backups, and
instance control (start/stop/restart).

The design should feel like a modern, premium cloud SaaS dashboard — think
Vercel, Railway, DigitalOcean, or Render. Clean, spacious, professional.
Use Bootstrap 5 as the CSS framework (Odoo 18 bundles it). Use Font Awesome 5
for icons.

IMPORTANT: The homepage at "/" is currently empty and has no content.

PAGE 1: SERVICES CATALOG
Path: /services
Auth: Public (no login required)
Template ID: services_page
--------------------------------------------------------------------------------

PURPOSE: The main storefront. This is where visitors discover what services are
available. It acts as the entry point to the sales funnel.

LAYOUT & CONTENT:
- Hero section at the top with a headline like "Cloud Business Solutions" and a
  short tagline explaining the platform's value proposition.
- Below the hero, a responsive grid of service cards (3 columns on desktop,
  2 on tablet, 1 on mobile).
- Each service card contains:
  * Service icon/image (stored as base64, rendered as <img>)
  * Service name (bold, prominent)
  * Subtitle (short descriptor, e.g. "Complete ERP Solution")
  * A bulleted feature list (from the service's feature_line_ids), showing
    3-5 key features with checkmark icons
  * Number of available plans (e.g. "4 plans available")
  * A primary CTA button: "View Plans" linking to /services/<product_id>
- If no published services exist, show a centered empty-state message:
  "No services available at the moment. Please check back soon." with a
  subtle illustration or icon.
- Cards should have subtle shadows, rounded corners, and hover effects
  (slight lift/shadow increase on hover).

DATA SOURCES:
- Services: saas.product records where is_published=True, ordered by sequence
- Features: saas.product.feature.line records linked to each product
- Plan count: count of related saas.plan records


--------------------------------------------------------------------------------
PAGE 2: SERVICE PLANS / DETAIL
Path: /services/<int:product_id>
Auth: Public
Template ID: service_plans_page
--------------------------------------------------------------------------------

PURPOSE: Shows all available plans for a specific service. This is where the
visitor compares pricing tiers and decides which plan to choose.

LAYOUT & CONTENT:
- Breadcrumb navigation: Services > [Service Name]
- Service header section:
  * Large service icon/image on the left
  * Service name (h1)
  * Subtitle
  * Full HTML description of the service
- Billing period toggle: A styled toggle/switch between "Monthly" and "Yearly"
  * When "Yearly" is selected, show a discount badge (e.g. "Save 20%")
  * The toggle dynamically switches all displayed prices without page reload
- Free Trial Banner (shown only if a trial plan exists AND the user has not
  already used a trial for this service):
  * Highlighted banner/card with a distinct background color (e.g. light blue
    or gradient)
  * Text: "Start with a Free Trial — No credit card required"
  * Shows trial plan specs (CPU, RAM, storage)
  * CTA button: "Start Free Trial" linking to the configure page with the
    trial plan. If user is not logged in, it links to /services/register
    with redirect parameters.
- Plan cards grid (3 columns desktop, responsive):
  * Each paid plan card includes:
    - Plan name (e.g. "Starter", "Professional", "Enterprise")
    - Monthly price displayed prominently (e.g. "$29/mo")
    - When yearly toggle is active: show yearly price per month with the
      yearly_discount_pct badge and a strikethrough on the monthly price
    - Specification list with icons:
      * CPU cores (fa-microchip)
      * RAM in MB/GB (fa-memory)
      * Storage in GB (fa-hdd)
      * Max backups (fa-shield-alt)
      * Recommended users (fa-users)
    - CTA button:
      * Logged-in users: "Get Started" → /services/<id>/plans/<plan_id>/configure
      * Not logged in: "Register to Get Started" → /services/register?...
  * Optionally highlight one plan as "Most Popular" or "Recommended" with a
    ribbon/badge

DATA SOURCES:
- Product: saas.product record by ID
- Plans: saas.plan records linked to this product, is_trial_plan=False
- Trial plan: saas.plan where is_trial_plan=True, linked to product
- Trial eligibility: check if current user has any existing trial instance for
  this product

PAGE 3: INSTANCE CONFIGURATION
Path: /services/<int:product_id>/plans/<int:plan_id>/configure
Auth: Required (redirects to login/register if not authenticated)
Template ID: service_configure_form
--------------------------------------------------------------------------------

PURPOSE: The final step before ordering. The user chooses a subdomain and domain
for their new instance.

LAYOUT & CONTENT:
- Two-column layout on desktop (form left, summary right), stacked on mobile.
- LEFT COLUMN — Configuration Form:
  * Heading: "Configure Your Instance"
  * Form fields:
    - Subdomain (text input):
      * Label: "Choose your subdomain"
      * Placeholder: "my-company"
      * Live validation feedback below the input:
        - While typing (debounced 500ms): shows spinner "Checking..."
        - Available: green checkmark + "subdomain is available"
        - Taken: red X + "subdomain is already taken"
        - Invalid format: red X + format requirements message
      * Validation regex: ^[a-z0-9]([a-z0-9-]{0,61}[a-z0-9])?$
      * Calls /saas/check-subdomain JSON endpoint for availability
    - Domain (dropdown select):
      * Label: "Select your domain"
      * Populated with available saas.domain records
      * Shows the full URL preview: "your-subdomain.example.com"
  * Error messages area (displayed if redirected back with errors)
  * Submit button:
    - Trial plan: "Start Free Trial" (green/success color)
    - Paid plan: "Continue to Payment" (primary/blue color)
    - Disabled until subdomain is validated as available
- RIGHT COLUMN — Order Summary Card:
  * Sticky/fixed position on scroll
  * Service name and icon
  * Selected plan name
  * If trial: "Free Trial" badge prominently displayed
  * Plan specs (CPU, RAM, Storage, Backups)
  * Price (if paid plan): monthly or yearly based on selection
  * A reassurance line: "You can upgrade anytime"

DATA SOURCES:
- Product: saas.product by ID
- Plan: saas.plan by ID
- Domains: saas.domain records (active)

JAVASCRIPT:
- subdomain_check.js handles real-time subdomain validation
- Debounces input by 500ms, calls /saas/check-subdomain
- Toggles submit button enabled/disabled based on result

PAGE 4: REGISTRATION
Path: /services/register
Auth: Public
Template ID: registration_form
--------------------------------------------------------------------------------

PURPOSE: Two-step registration flow. Step 1 collects user info, Step 2 verifies
phone via OTP. After verification, the user is automatically logged in and
redirected to the configure page.

LAYOUT & CONTENT:
- Centered card layout (max-width ~500px), clean and minimal.
- Progress indicator showing Step 1 / Step 2.

STEP 1 — Registration Form:
  * Heading: "Create Your Account"
  * Subheading: "Get started in minutes"
  * Form fields (vertically stacked):
    - Full Name (text, required) — fa-user icon
    - Email (email, required) — fa-envelope icon
    - Phone (tel, required) — fa-phone icon
      * Validated per selected country's phone format
    - Company Name (text, optional) — fa-building icon
    - Country (select dropdown, required) — fa-globe icon
      * Populated from res.country records
    - City (text, required) — fa-map-marker-alt icon
    - Street Address (text, optional) — fa-road icon
    - Job Title (text, optional) — fa-briefcase icon
    - Password (password, required, min 8 chars) — fa-lock icon
      * Strength indicator bar (weak/medium/strong)
    - Confirm Password (password, required) — fa-lock icon
      * Match validation shown in real-time
  * Error display area: shows field-specific validation errors returned from
    the server (email taken, phone invalid, etc.)
  * Submit button: "Continue" or "Verify Phone Number"
  * Footer link: "Already have an account? Sign in"

STEP 2 — OTP Verification:
  * Heading: "Verify Your Phone Number"
  * Subheading: "We sent a 6-digit code to +XXX-XXX-XXXX"
  * OTP input: Either 6 individual digit boxes or a single input field
  * Countdown timer: "Code expires in 9:45" (10-minute expiry)
  * "Resend Code" button (enabled after countdown or immediately)
  * Submit button: "Verify & Create Account"
  * Error message area for invalid/expired OTP

POST-REGISTRATION FLOW:
  After OTP verification:
  1. Creates partner record with all contact info
  2. Creates portal user
  3. Auto-logs in the user
  4. Redirects to the configure page they came from (preserves product/plan
     context via query parameters)

DATA SOURCES:
- Countries: res.country records
- OTP: saas.registration.otp model

--------------------------------------------------------------------------------
PAGE 6: INSTANCE DETAIL (Portal)
Path: /my/instances/<int:instance_id>
Auth: Required (portal user, access token supported)
Template ID: portal_instance_detail
--------------------------------------------------------------------------------

PURPOSE: The main management page for a single instance. Shows status, resources,
billing, backups, and all available actions. This is the most complex page.

LAYOUT & CONTENT:
- Two-column layout (content left ~8 cols, sidebar right ~4 cols).
- Breadcrumb: Home > My Instances > [Instance Name]

LEFT COLUMN:

  A) Instance Information Card:
    * Instance name (h2)
    * State badge (color-coded, same scheme as list page)
    * Instance URL (clickable, opens in new tab) — only if running
    * Service name
    * Plan name
    * Domain (subdomain.domain.com)

  B) Resource Usage Card (shown only if state is running or stopped):
    * Three horizontal progress bars:
      - CPU Usage: percentage bar (green < 60%, yellow 60-80%, red > 80%)
        Label: "CPU: 25%"
      - Memory Usage: percentage bar with same color coding
        Label: "RAM: 512MB / 1024MB"
      - Storage Usage: percentage bar
        Label: "Storage: 2.5GB / 5GB"
    * "Refresh" button (fa-sync icon) that calls /my/instances/<id>/refresh-usage
      via AJAX and updates bars without page reload
    * Last updated timestamp

  C) Alert Banners (contextual, shown based on instance state):
    * TRIAL ACTIVE: Blue info alert
      "You are on a free trial. Trial ends on [date]. Upgrade to a paid plan
      to keep your instance running."
      Button: "Upgrade Now" → /my/instances/<id>/upgrade
    * TRIAL EXPIRED (suspended + trial): Red alert
      "Your trial has expired. Subscribe to a paid plan to restore access."
      Button: "Subscribe Now" → /my/instances/<id>/upgrade
    * PENDING PAYMENT: Yellow warning alert
      "Payment is required to activate your instance."
      Button: "Complete Payment" → /my/instances/<id>/checkout
    * SUSPENDED (non-trial): Orange alert
      "Your instance has been suspended due to non-payment. Contact support
      at [email] to resolve."
    * FAILED: Red alert
      "Instance setup failed. Please contact support at [email]."
    * CANCELLED: Gray alert
      "This instance has been cancelled."
      Button: "Start New Instance" → /services
    * PENDING UPGRADE: Yellow alert
      "You have a pending plan upgrade to [plan name]. Complete the payment
      to apply the upgrade."
      Button: "Complete Payment" → /my/instances/<id>/checkout
    * SCHEDULED DOWNGRADE: Blue info alert
      "Your plan will be downgraded to [plan name] on your next billing date."
      Button: "Cancel Downgrade" (POST form)
    * PROVISIONING: Blue alert with animated spinner
      "Your instance is being set up. This usually takes a few minutes..."

  D) Billing Section:
    * Heading: "Invoices"
    * Table of invoices:
      - Invoice Reference (link to Odoo portal invoice page)
      - Date
      - Amount (formatted with currency symbol)
      - Payment Status badge:
        * Paid: green
        * Not Paid: red
        * In Payment: yellow
        * Reversed/Cancelled: gray
      - Actions column:
        * "Cancel" button for unpaid invoices (with confirmation dialog)
        * Not shown for mandatory invoices (Renewal:, Data restoration:)
    * Empty state: "No invoices yet"

  E) Backups Section (shown if running OR has existing backups):
    * Heading: "Backups"
    * "Create Backup" button:
      - Disabled if: backup currently in progress, backup limit reached,
        or instance is on trial plan
      - Shows tooltip explaining why it's disabled
    * Backup limit indicator: "2 / 5 backups used"
    * Trial notice: "Backups are not available on the trial plan"
    * Backups table:
      - Backup name/date
      - Status badge (Running with spinner, Completed green, Failed red)
      - "Restore" button (with confirmation dialog, only for completed backups
        on running/stopped instances)
    * Auto-refresh: If any backup has state=running, the section auto-refreshes
      every 30 seconds (up to 10 minutes)

RIGHT COLUMN (Sidebar):

  A) Actions Card:
    * List of available action buttons (shown/hidden based on state):
      - "Open Instance" (fa-external-link-alt) — external link, if running
      - "Restart" (fa-redo) — AJAX call with confirmation, if running
      - "Stop" (fa-stop) — AJAX call with confirmation, if running
      - "Start" (fa-play) — AJAX call with confirmation, if stopped
      - "Upgrade Plan" (fa-arrow-up) — link, if trial + running/suspended
      - "Change Plan" (fa-exchange-alt) — link, if paid + running/stopped/suspended
      - "Complete Payment" (fa-credit-card) — link, if pending upgrade invoice
      - "Start New Instance" (fa-plus) — link, if cancelled
    * Actions use portal_actions.js for AJAX calls with loading spinners

  B) Plan Details Card:
    * Current plan name
    * Specs list with icons:
      - CPU (fa-microchip): "2 vCPU"
      - RAM (fa-memory): "2048 MB"
      - Storage (fa-hdd): "10 GB"
      - Backups (fa-shield-alt): "5 backups"
    * Next billing date (if applicable)
    * Billing period: "Monthly" or "Yearly"

JAVASCRIPT BEHAVIOR:
- portal_actions.js: Handles restart/stop/start buttons via AJAX. Shows
  confirmation dialog, loading spinner, then reloads on success or shows
  error alert on failure.
- Auto-polling: If instance state is provisioning, pending_payment, paid, or
  has a pending_plan_id, the page polls /my/instances/<id>/status every 3
  seconds. When the state changes, the page reloads. Polling stops after 10
  minutes maximum.
- Backup auto-refresh: If a backup is in "running" state, the backup section
  refreshes every 30 seconds.

DATA SOURCES:
- Instance: saas.instance record (with all computed fields)
- Invoices: account.move records linked to this instance
- Backups: saas.instance.backup records for this instance
- Plan: saas.plan record

================================================================================
                        API / JSON ENDPOINTS
================================================================================

These are not full pages but AJAX endpoints used by the frontend JavaScript:

1. POST /saas/check-subdomain (JSON)
   Request: { subdomain: string, domain_id: int }
   Response: { available: bool, message: string }

2. POST /my/instances/<id>/status (JSON)
   Response: { state: string, ... instance status dict }

3. POST /my/instances/<id>/restart (JSON)
   Response: { success: bool } or { error: string }

4. POST /my/instances/<id>/stop (JSON)
   Response: { success: bool } or { error: string }

5. POST /my/instances/<id>/start (JSON)
   Response: { success: bool } or { error: string }

6. POST /my/instances/<id>/refresh-usage (JSON)
   Response: { cpu_usage, cpu_pct, ram_percent, ram_pct, total_storage,
               storage_pct, storage_limit }


================================================================================
                        DESIGN GUIDELINES
================================================================================

- Framework: Bootstrap 5 (included with Odoo 18)
- Icons: Font Awesome 5
- Color palette: Modern and clean. Use a primary blue for CTAs, green for
  success/running states, red for errors/failed, yellow/orange for warnings,
  gray for inactive/cancelled.
- Typography: Clean sans-serif. Large headings, readable body text.
- Cards: Rounded corners (border-radius: 8-12px), subtle box shadows,
  white backgrounds. Hover effects on interactive cards.
- Buttons: Rounded, clear hierarchy (filled primary, outline secondary).
- Spacing: Generous whitespace. Don't crowd elements.
- Responsiveness: Must work well on desktop (1200px+), tablet (768px), and
  mobile (< 576px).
- Status badges: Pill-shaped, color-coded, with optional icons.
- Progress bars: Bootstrap progress bars with dynamic color based on usage
  percentage thresholds.
- Forms: Clean, labeled inputs with validation feedback (green check / red X).
- Tables: Striped or hover-highlighted rows, clean headers.
- Animations: Subtle transitions on hover, loading spinners for async actions.
- Overall feel: Enterprise-grade but approachable. Similar to Vercel, Railway,
  DigitalOcean, or Render dashboards.
