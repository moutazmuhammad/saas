1. WHAT THIS MODULE DOES
================================================================================

This is a full SaaS platform manager built as an Odoo 18 module. It allows you
to provision, manage, and monitor multi-tenant Odoo instances running in Docker
containers on remote servers -- all from within the Odoo UI.

Key capabilities:
- Provision Odoo instances as Docker containers on remote servers via SSH
- Full lifecycle: deploy, start, stop, restart, suspend, redeploy, delete
- Automatic PostgreSQL database creation on separate DB servers
- Nginx reverse proxy with automatic SSL certificates (Certbot)
- Module marketplace: fetch modules from Docker images, organize into products
- Custom Git repositories: clone, pull, scan for modules
- Cloud backups to S3, Google Cloud Storage, or DigitalOcean Spaces
- Resource monitoring: CPU, RAM, disk, DB size
- Storage limit enforcement with automatic suspension
- Live Docker log streaming via Server-Sent Events
- Sales integration: modules and products are standard Odoo products

3.1  saas.instance  (saas_instance.py)  --  THE MAIN MODEL
--------------------------------------------------------------------------------

The central model. Represents one Odoo instance = one Docker container + one

FIELDS:

    url              Char      Computed: https://name
    plan_id          M2O       -> saas.plan

  Infrastructure:
    odoo_version_id  M2O       -> saas.odoo.version (Docker image + version)
    docker_server_id M2O       -> saas.container.physical.server
    db_server_id     M2O       -> saas.psql.physical.server
    xmlrpc_port      Integer   Host port for Odoo HTTP (auto-assigned)
    longpolling_port Integer   Host port for websocket (auto-assigned)

  Credentials (generated automatically):
    admin_password   Char      Odoo master password
    db_user          Char      PostgreSQL role
    db_password      Char      PostgreSQL password

  Modules:
    module_line_ids        O2M  -> saas.instance.module.line (all install requests)
    product_line_ids       O2M  -> saas.instance.module.line (filtered: products only)
    single_module_line_ids O2M  -> saas.instance.module.line (filtered: modules only)
    installed_module_ids   M2M  -> product.product (successfully installed)

  Resource Usage (updated by cron every 10 min):
    cpu_usage          Char    e.g. "2.35%"
    ram_usage          Char    e.g. "256MiB / 1GiB"
    ram_percent        Char    e.g. "25.00%"
    disk_usage         Char    e.g. "1.2 GB"
    db_size            Char    e.g. "500 MB"
    total_storage      Char    disk + db combined
    usage_last_updated Datetime

  Sales & Invoicing:
    sale_order_id      M2O       -> sale.order (linked sale order)
    sale_order_count   Integer   Computed
    invoice_count      Integer   Computed from sale_order_id.invoice_ids

  Other:
    state              Selection  draft/pending_payment/paid/provisioning/
                                  running/stopped/failed/suspended/cancelled
    provisioning_log   Text       Timestamped operation log
    extra_config       Text       Additional odoo.conf key=value lines
    repo_ids           O2M        -> saas.instance.repo (custom Git repos)
    backup_ids         O2M        -> saas.instance.backup

STATE MACHINE (CLIENT LIFECYCLE):

    The lifecycle is divided into two phases: billing and infrastructure.

    Billing phase (when a plan is assigned):
      draft --[Confirm & Bill]--> pending_payment --[payment received]--> paid
                                  pending_payment --[Mark as Paid]------> paid

    Infrastructure phase:
      paid --[Deploy]--> provisioning --[success]--> running
                                      --[error]---> failed

    Without a plan (internal/test instances):
      draft --[Deploy]--> provisioning --> running

    Operational transitions:
      running --[stop]------> stopped
      running --[suspend]---> suspended
      running --[redeploy]--> provisioning --> running
      stopped/suspended --[restart]--> running
      any (except provisioning) --[delete]--> cancelled

    Reset paths:
      failed/cancelled/pending_payment/paid --[Reset to Draft]--> draft

KEY METHODS:
    -- Billing --
    action_confirm_and_bill()  Create SO, confirm, generate & post invoice,
                               transition to pending_payment (or paid if $0)
    action_mark_as_paid()      Manual override: pending_payment -> paid
    action_view_sale_order()   Open linked sale order
    action_view_invoices()     Open invoices from linked sale order

    -- Deployment --
    action_deploy()          Validate + queue background deployment.
                             Requires paid state (or draft without plan).
    _do_deploy()             Create dirs, provision PG, render configs,
                             start container, setup Nginx+SSL

    -- Operations --
    action_stop()            Queue background stop
    action_restart()         Queue background restart
    action_suspend()         Queue background suspend
    action_redeploy()        Queue background redeploy (clone repos, pull,
                             update configs, install modules, restart)
    action_delete_instance() Queue background deletion (compose down, rm dirs,
                             drop PG, delete backups, remove nginx)
    action_create_backup()   Queue background backup
    action_config()          Read odoo.conf and show in popup
    action_docker_compose()  Read docker-compose.yml and show in popup
    action_view_logs()       Open live log stream (SSE)
    action_refresh_usage()   Fetch CPU/RAM/disk/DB size (synchronous)
    _cron_refresh_usage()    Cron: refresh usage for all running instances
    _cron_check_storage_limits()  Cron: suspend instances over storage limit

BACKGROUND PATTERN:
    All long-running SSH operations run in background threads:
    1. Action method validates + sets state to 'provisioning'
    2. Calls run_in_background() which uses postcommit to start thread
    3. Thread opens its own DB cursor, executes work, commits
    4. On error: rolls back, opens new cursor, sets error state

3.3  account.move (inherited)  (account_move.py)
--------------------------------------------------------------------------------

Detects invoice payment and auto-transitions linked SaaS instances.

OVERRIDES:
    write()  Watches for payment_state changes to 'paid' or 'in_payment'.
             When detected, finds SaaS instances linked via sale_order_id
             that are in 'pending_payment' state and transitions them to
             'paid'. Posts a chatter message on each instance.

HOW IT WORKS:
    1. Customer pays the invoice (via bank, online, or manual reconciliation)
    2. Odoo sets payment_state = 'paid' on the account.move
    3. Our write() override catches this change
    4. Searches sale.order records linked to the paid invoices
    5. Searches saas.instance records linked to those sale orders
    6. Transitions matching instances: pending_payment -> paid
    7. Immediately calls action_deploy() to provision the instance
    8. The client receives their running instance without any manual step

10. KEY WORKFLOWS
================================================================================

CLIENT LIFECYCLE (END-TO-END):

  The full flow from plan selection to a running instance:

  STEP 1 -- CONFIGURE (state: draft)
    a. Create a new instance record
    b. Fill in: subdomain, base domain, customer, plan, Odoo version,
       Docker server, DB server
    c. Add product/module lines (what to install on the instance)

  STEP 2 -- BILLING (state: draft -> pending_payment)
    a. Click "Confirm & Bill" button
    b. System validates: customer, plan, plan's product are all set
    c. Creates a sale order with:
       - Plan product as the main line (plan price)
       - Any selected SaaS products with a price as additional lines
    d. Auto-confirms the sale order
    e. Generates and posts the invoice (state = open)
    f. Instance transitions to "pending_payment"
    g. If invoice amount is zero: skips to "paid" automatically

  STEP 3 -- PAYMENT & AUTO-DEPLOY (state: pending_payment -> paid -> provisioning)
    Option A -- Normal payment (fully automatic):
      a. Customer pays the invoice (bank transfer, online payment, etc.)
      b. Odoo reconciles the payment -> invoice payment_state = 'paid'
      c. account.move.write() override detects the change
      d. Finds linked SaaS instance via sale_order_id
      e. Transitions instance to "paid" state
      f. Immediately triggers action_deploy() -- no manual step needed
      g. Instance goes to "provisioning" and then "running" automatically

    Option B -- Manual override (managers only):
      a. Click "Mark as Paid" button (for wire transfers, trials, etc.)
      b. Instance transitions to "paid" and auto-deploys immediately

    Zero-amount invoices:
      Handled in Step 2 -- if invoice total is $0, the instance skips
      pending_payment entirely and auto-deploys right after billing.

  DEPLOYMENT DETAILS (state: paid -> provisioning -> running)
    Triggered automatically by payment. Background thread executes:
       - Auto-assigns ports (finds unused pair starting from config port)
       - Generates random credentials (db_user, db_password, admin_password)
       - Creates directory structure on Docker server via SSH
       - Creates PostgreSQL role + database on DB server via SSH
       - Renders docker-compose.yml and odoo.conf from Jinja2 templates
       - Starts container: docker compose up -d
       - Waits for Odoo to be ready (polls HTTP for 120s)
       - Installs base modules via docker compose run
       - Installs selected modules (from product/module lines)
       - Obtains SSL certificate via Certbot
       - Deploys Nginx reverse proxy config
       - Sets state = "running"
       - Auto-refreshes resource usage
    On failure: state = "failed", Deploy button appears for manual retry

  INTERNAL/TEST INSTANCES (no billing):
    If no plan is assigned, the Deploy button is available directly from
    draft state -- the billing flow is skipped entirely.

  RESET PATHS:
    From pending_payment, paid, failed, or cancelled:
      Click "Reset to Draft" to return to draft and start over.

INSTANCE DEPLOYMENT (DETAILED STEPS):
  Same as Step 4 above. The background thread:
  1. Creates dirs: {base_path}/{partner_code}/{subdomain}/
  2. Provisions PostgreSQL role + database
  3. Renders configs from Jinja2 templates
  4. Runs docker compose up -d
  5. Waits for readiness
  6. Installs modules
  7. Sets up SSL + Nginx
  8. Sets state = "running"

MODULE INSTALLATION (on running instance):
  1. Add new product/module lines to instance
  2. Click "Redeploy"
  3. Background thread:
     a. Clones any pending custom repos
     b. Pulls all cloned repos
     c. Updates docker-compose.yml (adds new volume mounts)
     d. Updates odoo.conf (adds new addons paths)
     e. Installs pending modules via docker compose run
     f. Restarts container

CUSTOM REPOSITORY (instance-level):
  1. Add repo in instance "Custom Repos" tab (URL, branch, token)
  2. Click "Clone" -> clones repo on Docker server via SSH
  3. Click "Pull" -> git pull latest changes
  4. Redeploy instance to pick up new modules

CUSTOM REPOSITORY (version-level, via Product form):
  1. Create a Product, fill in Repository URL
  2. Click "Clone Repo" -> creates saas.version.repo, clones on server
  3. Click "Fetch Modules & Icons" -> scans repo for application modules,
     creates product.template records for each found module
  4. The product can now be installed on any instance of that version

BACKUP FLOW:
  1. Manual: click "Create Backup" on instance
  2. Automatic: daily cron for all running instances
  3. Process:
     a. pg_dump database on DB server
     b. Tar instance folder on Docker server
     c. Upload to configured cloud storage (S3/GCS/DO Spaces)
     d. Store presigned download URL (7-day expiry)
  4. Old backups auto-cleaned based on plan's max_backups

11. BACKGROUND PROCESSING PATTERN
================================================================================

All long-running SSH operations use a common pattern to avoid blocking the UI:

UTILITY: run_in_background()  (in utils.py)

How it works:
  1. The action method validates inputs and sets intermediate state
  2. Calls run_in_background(record, '_do_method', error_method, ...)
  3. run_in_background uses env.cr.postcommit to ensure the current
     transaction is committed BEFORE the thread starts
  4. A daemon thread opens its own DB cursor
  5. Executes the work method on a fresh record instance
  6. On success: commits
  7. On failure: rolls back, opens new cursor, calls error handler
     (typically reverts state and logs the error)

Operations using this pattern:
  - Deploy, Redeploy
  - Stop, Restart, Suspend
  - Delete Instance
  - Create Backup
  - Clone/Pull repos (instance and version level)
  - Fetch modules from Docker image
  - Fetch modules from custom repo
